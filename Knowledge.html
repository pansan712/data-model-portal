📖 Concept Deep Dives

Dimensional Modeling (Kimball Methodology) ▼

Dimensional modeling, pioneered by Ralph Kimball, organizes data into fact tables (measurements/metrics) and dimension tables (context/descriptors). This creates a star-shaped schema optimized for analytical queries.


Key Principles:

  • Conformed dimensions shared across fact tables enable cross-process analysis
  • Surrogate keys decouple from source systems and enable SCD tracking
  • Grain definition is critical – one row = one measurement event
  • Degenerate dimensions (order numbers) live in the fact table
  • Bridge tables resolve many-to-many relationships
Data Vault 2.0 ▼

Data Vault 2.0 is a detail-oriented, historical tracking, and uniquely linked set of normalized tables that support one or more functional areas of business. It uses three core entity types:


Core Components:

  • Hubs – Business keys (unique identifiers of business concepts)
  • Links – Associations/relationships between hubs
  • Satellites – Descriptive attributes with full history

Key Features:

  • SHA-256 hash keys for deterministic, parallel loading
  • All timestamps in UTC for consistency
  • Record source tracking in every satellite
  • Additive architecture – never alter existing structures
Activity Schema Pattern ▼

Activity Schema is a modern approach that models all business events as a single, wide activity stream. Each row represents something that happened (an activity) with associated entity references and attributes.


Structure:

  • Activity Stream – Single wide table with: activity_id, entity_id, activity_type, timestamp, and feature columns
  • Entity Table – Enrichment data for the primary entity (customer, user, account)
  • All analytics computed by filtering and aggregating the activity stream

Benefits:

  • Simple schema – just 2 tables for most use cases
  • Natural event ordering and temporal queries
  • Easy to add new activity types without schema changes
  • Works well with modern columnar storage (BigQuery, Snowflake)
Graph Data Models ▼

Graph models represent data as nodes (entities) and edges (relationships), ideal for highly connected data where relationships are as important as the entities themselves.


Use Cases:

  • Supply chain & logistics networks
  • Fraud detection & financial crime
  • Social network analysis
  • Knowledge graphs & recommendation engines
  • Impact analysis & data lineage

Technologies:

  • Neo4j (native graph database)
  • Apache TinkerPop / Gremlin
  • BigQuery graph analytics (SQL-based)
  • Databricks GraphFrames
Naming Conventions Standard ▼

Consistent naming is critical for maintainability, collaboration, and automated tooling. Our standards:


Table Prefixes:

  • dim_ – Dimension tables
  • fact_ – Fact tables
  • hub_, link_, sat_ – Data Vault entities
  • stg_ – Staging tables
  • int_ – Intermediate/transform tables
  • mart_ – Mart/presentation tables

Column Suffixes:

  • _key, _sk – Surrogate keys
  • _nk – Natural keys
  • _date – Date columns
  • _at, _ts – Timestamps
  • _flag, _ind – Boolean indicators
  • _amt – Monetary amounts
  • _qty – Quantities
  • _cnt – Counts

Required Audit Columns (all tables): created_at, updated_at, created_by, updated_by, is_deleted
